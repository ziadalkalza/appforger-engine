# Reconciliation Record Policy

A reconciliation record proves that related outputs from different agents, providers, tools, or branches were compared and resolved.

Use a reconciliation record when:
- cross-brand dependent work occurs
- backend and frontend outputs must be aligned
- a mock contract is replaced by a real API baseline
- two parallel outputs affect the same feature
- an agent output conflicts with a baseline

A reconciliation record must include:
- reconciliation ID
- reason/trigger
- affected agents/providers/brand families
- artifacts checked
- mismatches found
- resolution selected
- final source-of-truth files
- approver or reviewer
- status

Reconciliation does not replace approval. It documents consistency review.
