# Test Case Writing Rules

A test case must include:

- Test ID
- Feature/screen/API reference
- Requirement or acceptance criterion
- Preconditions
- Steps
- Expected result
- Test type
- Automation target
- Data needed
- Baseline dependency
- Evidence required
- Owner/status

Good tests cover:

- success path
- validation errors
- empty/loading/error states
- permissions/auth failures
- network/backend failures
- edge cases
- accessibility
- visual correctness when UI is affected
- regression risk

Do not create vague tests such as “test login works.” Instead define exact inputs, actions, expected results, and evidence.
